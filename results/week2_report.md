**Verdict:** Occasion embeddings correctly encode the formality spectrum. Retrieval queries built from these vectors should distinguish between, e.g., casual daytime fragrances and black-tie-appropriate ones.

## 3. Raw vs Enriched Retrieval — Side-by-Side Top-5

> **Pending:** `artifacts/retrieval_comparison.txt` — being generated by pipeline run.

### What to look for

The `embedding_text` column (raw) is Karan's note concatenation:
```
bergamot, lemon | jasmine, rose | sandalwood, vanilla
```
This is ingredient-level and cannot bridge the semantic gap to occasion descriptions like "formal charity gala at night with a polished, luxurious mood."

The `retrieval_text` column (enriched) uses LLM-generated structure:
```
Brand: Creed | Name: Aventus | Accords: Woody, Citrus
Top: Bergamot, Blackcurrant | Heart: Birch, Pineapple | Base: Musk, Oak
Season: fall | Best for: formal event
Formality: high | Character: bold, sophisticated, powerful
Vibe: A confident evening statement for sharp tailoring and high-stakes occasions.
```

The enriched format gives the embedding model season, occasion, formality level, and a complete vibe sentence — all of which appear verbatim in the query-side occasion descriptions. We expect meaningfully different top-5 sets, especially for the formality extremes (`casual_day` vs `black_tie`).

## 5. Model Justification

### Text Embedding — `gemini-embedding-001`

- MTEB English rank #1 at 68.32 — best available text embedding model from the Gemini API
- 1536-dimensional output; free-tier accessible
- Used with `RETRIEVAL_QUERY` for occasions, `RETRIEVAL_DOCUMENT` for fragrances

### Multimodal Embedding — `Qwen3-VL-Embedding-8B`

Replaces the original plan's `gemini-embedding-2`.

| Model | MMEB-V2 Score | Availability | Notes |
|---|---|---|---|
| `Qwen3-VL-Embedding-8B` | **77.8** (#1 overall) | Local GPU (16 GB) | VLM backbone, native multimodal |
| `gemini-embedding-2` | 68.9 | Google API | Dual-encoder, API quota risk |

The 9-point gap on MMEB-V2 — the cross-modal retrieval benchmark directly relevant to outfit→fragrance retrieval — justifies the GPU dependency. The model runs locally on Colab Pro A100, eliminating API quota risk during the demo. Implementation is vendored from the HuggingFace model repo at `src/vibescents/qwen3_vl_embedding.py`.

### Enrichment Model — `gemini-3-flash-preview`

Low-cost structured output generation. Produces 7 enriched fields per fragrance row via JSON schema enforcement. The schema forces discrete season/occasion values and bounded 0.0–1.0 scores for formality, fresh_warm, and day_night, preventing free-form drift.

### Reranker — `gemini-3.1-pro-preview`

Multimodal input (outfit image + occasion text + candidate retrieval_text + structured metadata). Returns per-candidate scores in four sub-dimensions (overall, formality, season, freshness) plus a short rationale. Research scope only — does not replace the baseline unless it wins on the benchmark.

### Evaluation Judge — `gemini-2.5-pro`

Deliberately different model from the label generator (`gemini-3.1-pro-preview`) to avoid self-confirmation bias.

## 7. Failure Analysis

> **Pending:** requires retrieval comparison output.

### Known failure modes to check

1. **Enrichment field gaps**: Rows where the Gemini Flash model returned empty `vibe_sentence` or generic `likely_occasion`. These degrade retrieval_text quality silently.
2. **Note-only fragrances**: Rows in `vibescent_500.csv` where `main_accords` is null — the retrieval_text falls back to notes only, which is closer to raw baseline quality.
3. **Occasion ambiguity**: `streetwear_night` and `casual_day` share considerable semantic overlap (cosine 0.656). Any fragrance that scores well for one may also rank for the other — this tests whether formality differentiation is working.
4. **Formality floor**: Very casual fragrances (low formality score) should rank low for `black_tie`. If they don't, the enriched formality field is not being captured by the embedding.
